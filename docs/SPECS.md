# シフト作成支援ツール 仕様書 (What/Why)

## 1. アプリの目的 (Why)
ある業務において、平日夜間・休日昼間・休日夜間などを対象としたシフト表を作成する。
シフト作成には複雑なルールが複数存在し、すべてを自動で満たそうとすると現場の細かな事情（急な予定変更、人間関係など）に対応しきれない。そのため、完全な自動生成ではなく「人間の作成作業を支援・検証するツール」として機能することを目的とする。

## 2. 対象となる対応日 (What)
対応が必要な枠は以下の通り定義する。

* 平日: 夜間
* 土曜: 昼間 / 夜間
* 日曜・祝日: 昼間 / 夜間
* 長期連休 (GW、シルバーウィーク、年末年始等): 昼間 / 夜間

## 3. 対応者の属性とルール (What)
シフトにアサインされる対応者は、以下の属性を持つ。

* **所属課**: A課, B課, C課のいずれかに所属
* **スキルランク**: A, B, C, Dのいずれかを付与
* **雇用形態**: 一般、または特別雇用者（平日夜間のみ対応可能）
※特別雇用者についても、一般と同様に所属課とスキルランクが付与される。

## 4. シフト表作成ルール
シフト作成時にシステムが検証・判定・支援するルール。

### 4.1. 必須ルール (Error: 違反している場合はシフトを確定できない)
1. **人数・枠制限**: 1枠につき必ず2名1組で対応すること。また、1人あたり1日1枠とし、昼夜の通し勤務は不可とする。
2. **所属制限**: 同じ所属課の者を1組にしてはいけない。(例: A課同士はNG)
3. **スキル制限**: 1組の中に、必ず「スキルランクA」の者を1名以上含めること。
4. **勤務間隔**: 同一人物の対応日は、中9日以上空けなければならない。（例: 1日に入った場合、次回は11日以降）
5. **特別雇用者の枠制限**: 特別雇用者は「平日夜間」の枠のみアサイン可能とする。

### 4.2. 任意ルール (Warning: 違反していても警告のみで確定可能)
1. **休日の連続回避**: 同一人物に対して、連続して「平日夜間以外 (＝休日昼・夜、長期連休)」を設定しないよう配慮する。
2. **長期連休の負荷分散**: 前年に長期連休の対応を行った者は、当年の長期連休対応者に設定しないよう配慮する。


テナントごとにルールが変わることを見据えた設計、まさにSaaSアーキテクチャの急所を押さえていますね！
ハードコードを避け、初期段階から「ルールの可変性」を担保しておくことで、新しい顧客が入った時のオンボーディングが劇的に楽になります。

「現時点ではJSON形式で保持し、最終的にはDB側に寄せたい（専用のテーブルやカラムで正規化したい）」という方針に基づき、MVPにおけるベストプラクティスを整理しました。

PostgreSQL（Neon）を採用されているので、**DBの `tenants` テーブルに `rules` というJSONB型のカラムを持たせてしまう**のが、MVPとして最もスマートで実装コストが低いアプローチです。これならファイル管理の煩雑さ（デプロイしないとルールを変えられない問題）を避けつつ、DBの正規化は後回しにできます。

もし「DBには一切入れず、リポジトリ内の `rules.json` ファイルで管理する」という意図でしたら、バックエンドの起動時にメモリに読み込む形になります。

今回は、API連携のしやすさから「JSON構造の定義」として仕様をまとめました。

## 5. テナントごとのルール管理 (MVP方針)

本システムはSaaSとしての展開を前提としており、テナント（利用組織）ごとにシフト作成の制約（中9日ルール、スキルランク必須条件など）が異なる可能性が高い。

そのため、MVP（Minimum Viable Product）の段階では、ルールをアプリケーションコードにハードコードせず、**JSON形式のスキーマ定義**として保持・適用するアーキテクチャとする。

### 5.1. ルールJSONのスキーマ例
テナントごとに以下のようなJSON構造を持たせ、フロントエンドとバックエンドの双方がこの値を参照してバリデーションを実行する。

```json
{
  "shift_rules": {
    "min_interval_days": 10,           // 勤務間隔の必須日数（0なら制限なし）
    "require_skill_ranks": ["A"],      // ペアに最低1人必要なスキルランク
    "allow_same_department": false,    // 同じ所属課同士のペアを許可するか
    "special_employment_shifts": ["weekday_night"] // 特別雇用者が入れる枠
  },
  "warnings": {
    "avoid_consecutive_holidays": true // 休日連続アサインを警告するか
  }
}
```

### 5.2. システム間のルール共有
1. バックエンド (FastAPI) は、テナントに紐づく上記JSONを保持する。
2. フロントエンド (Next.js) は、アプリ初期ロード時またはシフト作成画面への遷移時に、バックエンドから `GET /api/tenants/me/rules` のようなエンドポイント経由でこのJSONを取得する。
3. フロントエンド・バックエンドともに、このJSONの値を参照してルールの計算処理を行う。

### 5.3. 将来の展望
MVPフェーズの検証完了後、設定UIの拡充などに伴い、このJSON構造を解体し、データベースの専用テーブル（`tenant_shift_rules` 等）へ正規化して移行する計画とする。
