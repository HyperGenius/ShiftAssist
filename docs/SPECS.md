# シフト作成支援ツール 仕様書 (What/Why)

## 1. アプリの目的 (Why)
ある業務において、平日夜間・休日昼間・休日夜間などを対象としたシフト表を作成する。

シフト作成には複雑なルールが複数存在し、すべてを自動で満たそうとすると現場の細かな事情（急な予定変更、人間関係など）に対応しきれない。

このアプリでは完全な自動生成ではなく「人間の作成作業を支援・検証するツール」として機能することを目的とする。

## 2. 対象となる対応日 (What)
対応が必要な枠は以下の通り定義する。

* 平日: 夜間
* 土曜・祝前日: 夜間
* 土曜: 昼間 / 夜間
* 日曜・祝日: 昼間 / 夜間
* 長期連休 (GW、シルバーウィーク、年末年始 等): 昼間 / 夜間

## 3. 対応者の属性とルール (What)
シフトにアサインされる対応者は、以下の属性を持つ。

* **所属課**: ユーザーが定義する課(A課, B課, C課...)のいずれかに所属
* **スキルランク**: ユーザーが定義するスキル(A, B, C, D)のいずれかを付与
* **雇用形態**: デフォルト(正職員)、特別雇用者（例: 平日夜間のみ対応可能）
  * 特別雇用者についても一般と同様に所属課とスキルランクを付与


## 4. シフト表作成ルール
シフト作成時にアプリで検証・判定・サジェストするルール。

### 4.1. 必須ルール (Error: 違反している場合はシフトを確定できない)
1. **人数・枠制限**: 1枠につき必ず2名1組で対応すること。また、1人あたり1日1枠とし、昼夜の通し勤務は不可とする。
2. **所属制限**: 同じ所属課の者を1組にしてはいけない。(例: A課同士はNG)
3. **リーダー適性**: 1組の中に、必ずリーダー適性者を1名以上含めること。
4. **勤務間隔**: 同一人物の対応日は、中9日以上空けなければならない。（例: 1日に入った場合、次回は11日以降）

### 4.2. 任意ルール (Warning: 違反していても警告のみで確定可能)
1. **休日の連続回避**: 同一人物に対して連続して「平日夜間以外 (＝休日昼・夜、長期連休)」を設定しないよう配慮する。
2. **長期連休の負荷分散**: 前年に長期連休の対応を行った者は、当年の長期連休対応者に設定しないよう配慮する。

## 5. テナントごとのルール管理 (MVP方針)

MVP（Minimum Viable Product）の段階では、ルールをアプリケーションコードにハードコードせず、**JSON形式のスキーマ定義**として保持・適用するアーキテクチャとする。

### 5.1. ルールJSONのスキーマ例
テナントごとに以下のようなJSON構造を持たせ、フロントエンドとバックエンドの双方がこの値を参照してバリデーションを実行する。

```json
{
  "shift_rules": {
    "min_interval_days": 10,           // 勤務間隔の必須日数（0なら制限なし）
    "require_skill_ranks": ["A"],      // ペアに最低1人必要なスキルランク
    "allow_same_department": false,    // 同じ所属課同士のペアを許可するか
    "special_employment_shifts": ["weekday_night"] // 特別雇用者が入れる枠
  },
  "warnings": {
    "avoid_consecutive_holidays": true // 休日連続アサインを警告するか
  }
}
```

### 5.2. システム間のルール共有
1. バックエンド (FastAPI) は、テナントに紐づく上記JSONを保持する。
2. フロントエンド (Next.js) は、アプリ初期ロード時またはシフト作成画面への遷移時に、バックエンドから `GET /api/tenants/me/rules` のようなエンドポイント経由でこのJSONを取得する。
3. フロントエンド・バックエンドともに、このJSONの値を参照してルールの計算処理を行う。

### 5.3. 将来の展望
MVPフェーズの検証完了後、設定UIの拡充などに伴い、このJSON構造を解体し、データベースの専用テーブル（`tenant_shift_rules` 等）へ正規化して移行する計画とする。

## 6. テナント固有の祝日・休日管理

### 6.1. 目的
シフト作成アルゴリズムにおいて「システム共通の祝日」ではなく「テナントごとのカレンダー」を基準に枠の種別を判定できるようにするための独立したデータ基盤。

### 6.2. データ定義 (`tenant_holidays` テーブル)

| カラム | 型 | 説明 |
|---|---|---|
| `id` | UUID | プライマリキー |
| `tenant_id` | String | テナント識別子（Clerk Organization ID） |
| `date` | Date | 休日の日付 |
| `name` | String | 休日名称（例: "元日"、"創立記念日"） |
| `is_long_holiday` | Boolean | 長期連休フラグ（GW・年末年始等） |
| `created_at` | DateTime | レコード作成日時 |

- `(tenant_id, date)` に一意制約 `uq_tenant_holiday_date` を設定し、重複登録をDBレベルで防止する。

### 6.3. REST API エンドポイント

| メソッド | パス | 説明 |
|---|---|---|
| `GET` | `/api/holidays/?year={year}&month={month}` | 一覧取得（年/月フィルタ対応） |
| `POST` | `/api/holidays/` | 休日を1件または複数件追加 |
| `DELETE` | `/api/holidays/{id}` | 指定した休日を削除 |

すべてのエンドポイントは `X-Tenant-Id` ヘッダーによるテナントアイソレーションが必須。

### 6.4. 日本標準祝日の自動シーディング

- `GET /api/holidays/?year={year}` を呼び出した際、対象年のデータが1件も存在しない場合に限り、`jpholiday` ライブラリを用いて日本の標準祝日を自動的にDBへ投入する。
- 自動投入されたデータは通常の休日と同様に編集・削除が可能。

### 6.5. 翌年祝日の取得制限

内閣府が翌年の祝日を例年9月頃に公開するため、以下の制限を設ける。

- **翌年への切り替えは実行月が9月（9月以降）のみ許可する。**
- **取得可能な年は最大で「現在年 + 1年」まで。**（内閣府が2年以上先の祝日を公開しないため）

フロントエンドでは年セレクターの「翌年」ボタンを上記条件に従い制御する。9月より前の月は翌年ボタンを非活性にし、ツールチップで理由を表示する。

### 6.6. フロントエンド管理画面

- URL: `/admin/settings/holidays`
- 機能: 年ナビゲーター付き休日一覧（削除可）+ 新規休日追加フォーム
- ローディング中は「内閣府ウェブサイトより祝日データ取得中」を表示する（`HolidayLoadingSkeleton` コンポーネント）。
- `HolidayLoadingSkeleton` は Storybook でプレビュー可能（`Holidays/HolidayLoadingSkeleton` ストーリー）。

## 7. シフト要件スナップショット保存仕様

### 7.1. 目的

フロントエンドのカレンダー表示で参照する「各枠の必要人数」を、将来的なルール変更が過去データに影響しないよう DB にスナップショットとして永続化する。

### 7.2. 読み出し仕様（過去データ表示時）

- `GET /api/shift-requirements?year={year}&month={month}` を呼び出すと、指定年月の `shift_requirements` テーブルのレコード一覧を JSON 配列で返す。
- DB 上にデータが存在する場合は **登録済みデータをそのまま返す**（ルール変更の影響を受けない）。

### 7.3. スナップショット保存仕様（新規作成時）

- バックエンドの `generate_requirements_for_month(session, tenant_id, year, month)` 関数を呼び出すことで、指定月の要件レコードを一括生成する。
- 生成ロジック:
  1. `tenant_holidays` テーブルから対象月の祝日・長期連休を取得する。
  2. 対象月の各日について、曜日・祝日フラグ・長期連休フラグを元に `SlotTypeEnum` を決定する。
  3. 既存レコードが存在しない `(date, slot_type)` の組み合わせのみ `required_headcount=2` (デフォルト) で INSERT する。
- 重複防止: `(tenant_id, shift_date, slot_type)` の一意制約 `uq_shift_req_tenant_date_slot` によりDB レベルでも二重登録を防止する。

### 7.4. 枠種別の決定ルール

| 日付の種別 | 生成される枠種別 |
|---|---|
| 長期連休（GW・年末年始等） | `long_hol_day` + `long_hol_night` |
| 土曜日 | `sat_day` + `sat_night` |
| 日曜日 or 祝日 | `sun_hol_day` + `sun_hol_night` |
| 平日（上記以外） | `weekday_night` のみ |

### 7.5. 過去データからのインポート（変換スクリプト）

- `scripts/convert_excel_to_csv.py` を実行して過去シフト Excel を変換する際、出力 CSV に `required_count` 列が付与される。
- `required_count` はその枠にアサインされたワーカー数（実績値）であり、過去データの `shift_requirements.required_headcount` としてそのまま利用できる。

### 7.6. REST API エンドポイント

| メソッド | パス | 説明 |
|---|---|---|
| `GET` | `/api/shift-requirements/?year={year}&month={month}` | 指定年月の要件一覧を取得 |
| `POST` | `/api/shift-requirements/` | 要件を1件作成 |
| `GET` | `/api/shift-requirements/{id}` | 指定した要件を取得 |
| `PUT` | `/api/shift-requirements/{id}` | 指定した要件を更新 |
| `DELETE` | `/api/shift-requirements/{id}` | 指定した要件を削除 |
| `PUT` | `/api/shift-requirements/{id}/assignments` | 指定した要件のアサイン情報を上書き保存 |

すべてのエンドポイントは `X-Tenant-Id` ヘッダーによるテナントアイソレーションが必須。
