# シフト作成支援ツール アーキテクチャ・技術スタック

本ドキュメントでは、シフト作成支援ツールの全体的なディレクトリ構成および採用する技術スタックについて定義する。

## 1. ディレクトリ構成

フロントエンドとバックエンドを分離し、インフラのコードも同一リポジトリで管理するモノレポ構成を採用する。

```text
/
├── backend/            # バックエンドコード (シフト検証ロジック / API提供 / DB操作)
│   ├── app/
│   │   ├── core/       # アプリケーションのコアロジック
│   │   ├── db/         # DB接続 (db.py)
│   │   ├── models/     # SQLModel (ORM) & Pydanticモデル
│   │   ├── routers/    # APIエンドポイント定義
│   │   └── services/   # ビジネスロジック
│   ├── alembic/        # DBマイグレーション
│   ├── scripts/        # バッチ処理 (run_batch.py), シードデータ
│   └── tests/          # 単体テスト & 統合テスト
│       ├── unit/       # ユニットテスト
│       └── integration/# 統合テスト
│
├── frontend/           # フロントエンドコード (UI / 状態管理 / API通信)
│   ├── src/
│   │   ├── app/        # Next.js App Router Pages
│   │   ├── components/ # UIコンポーネント
│   │   │   └── ui/     # カスタムSciFiコンポーネント (SciFiButton等)
│   │   ├── services/   # API呼び出し (api.ts)
│   │   ├── types/      # TypeScript型定義
│   │   └── utils/      # 汎用ユーティリティ
│   └── e2e/            # Playwrightテスト (準備中)
│
├── docs/               # プロジェクトドキュメント
│   ├── ARCHITECTURE.md # 本ドキュメント
│   └── ...             # 各種実装ガイド & レポート
│
└── infra/              # インフラストラクチャコード (Terraform等のSaaS・クラウド構成管理)
    └── neon/           # Neon DB作成
```

## 2. 技術スタック

複雑なシフト検証ロジックと、リッチで直感的なUI（ドラッグ＆ドロップ等の手動調整）を両立させるため、以下のモダンな技術スタックを採用する。

### フロントエンド
* **フレームワーク**: Next.js
* **ホスティング**: Vercel
* **役割**:
  * ユーザーインターフェースの提供
  * シフトの手動調整（ドラッグ＆ドロップなど）におけるクライアントサイドの状態管理
  * バックエンドAPIとの通信と、リアルタイムな警告・エラー表示

### バックエンド
* **言語/フレームワーク**: Python / FastAPI
* **データバリデーション**: Pydantic (BaseModel)
* **ホスティング**: Google Cloud Run (コンテナベースのサーバーレス環境)
* **役割**:
  * シフト作成ルールの厳密な検証（バリデーションロジックの本体）
  * データベースに対するCRUD操作
  * フロントエンド向けAPIの提供

### データベース
* **サービス**: Neon (サーバーレス PostgreSQL)
* **役割**:
  * ユーザー情報、対応者マスタ、シフトデータ、履歴等の永続化
  * ※バックエンドからORM等を用いて接続・操作を行う

### 認証・認可
* **サービス**: Clerk
* **役割**:
  * ユーザーのログイン、セッション管理
  * フロントエンド・バックエンド間の安全な認証フローの確立

## 3. システムの役割分担（フロントエンド vs バックエンド）

本システムでは、ユーザー体験（UX）の最大化とデータ整合性の担保を両立させるため、シフト作成時の計算とバリデーションについて以下の通り役割を明確に分離する。

### 3.1. フロントエンド (Next.js) の役割
**「アドホックな計算とリアルタイムなUIフィードバック」**

* **インメモリでの状態管理**:
  * 編集中のシフト表データは全てフロントエンドのメモリ（State）に保持する。
* **リアルタイム・バリデーション**:
  * ユーザーが対応者をアサイン（またはドラッグ＆ドロップ）した瞬間に、フロントエンド側でJavaScript/TypeScriptを用いてルール計算（中9日空き、スキルランク制限など）を実行する。
* **警告・エラーの即時表示**:
  * バックエンドと通信することなく、画面上に即座にアラート（「この組み合わせはランクAがいません」「中9日ルールに違反しています」など）を表示し、ユーザーの試行錯誤を阻害しない。
* **オーバーライド（強制確定）の意図の収集**:
  * ルール違反があってもユーザーが意図的に設定する場合（当人同士の合意など）、その「承諾フラグ」をデータとして持たせる。

### 3.2. バックエンド (FastAPI) の役割
**「最終検証（Source of Truth）とデータの永続化」**

* **確定時の厳密なバリデーション**:
  * フロントエンドから「保存・確定」リクエスト（JSON）を受け取った際、データベースの最新状態と照らし合わせて**再度全てのルールを検証**する。
  * これにより、複数人が同時に編集した際の競合（レースコンディション）や、不正なリクエストによるデータ矛盾を防ぐ。
* **オーバーライドの妥当性確認**:
  * ルール違反を含むリクエストの場合、「フロントエンド側で正しくオーバーライドフラグが付与されているか（意図的な違反か）」を確認し、付与されていなければ 400 Bad Request 等で弾く。
* **テナント分離の強制**:
  * 全ての操作が、認証されたユーザーの所属するテナント（Organization）内でのみ完結することを保証する。
* **トランザクション管理**:
  * 複数の枠（`shift_slots` と `shift_assignments`）の保存を1つのトランザクションで行い、途中でエラーが起きた場合は全てロールバックする。
